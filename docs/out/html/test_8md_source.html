1 # Plang
2 
3 This is the repository for the **Plang** language. The **Plang**
4 language is a mixture of several languages, including C and Python. It
5 is a simple translator that accepts **Plang** and translates it to C.
6 Some other iteration of the language may use LLVM to generate code. This
7 will never be an interpreted language system.
8 
9 ## Why?
10 
11 The reasons for implementing **Plang** are very simple. I love
12 programming languages and I love writing code. I have always wanted to
13 create a language from scratch and then write a bunch of code in it. I
14 have made numerous attempts, and as of February 2021, this is the latest
15 one.
16 
17 ## Goals
18 
19  - **Plang** is object oriented. Only classes are used to implement
20  functionality.
21  - **Plang** is strongly typed but also allows robust casting.
22  - **Plang** has garbage collection.
23  - **Plang** does not have pointers or NULL values.
24  - **Plang** supports maps, dictionaries, and lists natively.
25  - **Plang** has a simplified native type system that supports ints,
26  uints, floats, bools, and strings natively.
27  - **Plang** follows the notion of having as few keywords and operators
28  as possible, while still being able to implement complex programs.
29  It is intended to be a application language, rather than a system
30  language that has things like bitwise operations.
31  - **Plang** follows the notion of having a clear and obvious way to do
32  something. There may be more than one semantic paths, but that stems
33  from making a choice to put in optional elements or leave them out.
34  It's intended to be as simple as possible, but no simpler.

126 ## Constructors and Destructors
127 
128 Constructors and destructors, as one would expect, are called
129 automatically when a class is created or destroyed. There is a default
130 constructor and destructor that is always implemented for a class and
131 these are always called when a class is created or destroyed. There is
132 no need to declare the default methods, but they may be declared for
133 clarity. if desired.
134 
135 ### Constructor
136 
137 The default constructor accepts no parameters and it is always called
138 before any user-defined constructor in the same context. The default
139 constructor creates any inherited classes and applies default values to
140 any class attributes. For example, strings are initialized to zero
141 length and ints are given a value of zero. If a different value is
142 desired, then a constructor with no parameters can be declared and
143 implemented. This is treated the same as any other override constructor.
144 As with regular methods, the constructor may be overloaded by parameter
145 types. Overloaded constructors must be declared in the class
146 declaration. These constructors must be called explicitly but providing
147 parameters to the creation call to the class. For example:
148 
149 import imp1;
150 
151 class class1(imp1.clas0) {
152 
153 public int var1;
154 
155 constructor(int);
156 
157 }
158 
159 class1.constructor(var) {
160 
161 var1 = var;
162 
163 imp1.clas0.constructor(123, "a string");
164 
165 }
166 
167 class class2 {
168 
169 public int meth();
170 
171 }
172 
173 class2.meth() {
174 
175 class1 cls = class1(0x123);
176 
177 return(0);
178 
179 }
180 
181 In the above example, the default constructor for `class1` is called,
182 which sets `var1` to zero, then the override is called, which sets
183 `var1` to `0x123` after casting it to an `int`. When an override
184 constructor is declared, but not implemented, a syntax error is
185 generated.
186 
187 ### Destructor
188 
189 The default destructor is normally all that is required, since the
190 garbage collector should destroy any unreachable memory. However, a
191 destructor may be declared and implemented. This destructor will be
192 called after the default constructor as one would expect. If there is a
193 declaration for a destructor but there is no implementation, then a
194 warning is issued. Destructors accept no parameters and return no value,
195 so they may not be overridden. Multiple declarations or implementations
196 of destructors are a syntax error.

198 ## Exception Handling
199 
200 Exceptions are simplified and only allow for a string to be sent to the
201 handler. The `raise` keyword accepts a single parameter of a formatted
202 string. This is aimed at reporting errors more than actually trying to
203 fix them. So exceptions in **Plang** are an error handling mechanism and
204 not a flow control mechanism. All of the usual signals in a UNIX-like
205 environment are handled as exceptions and a reasonable string string is
206 generated for them. For things like errors when opening files, the
207 runtime code generates an exception and an appropriate string and raises
208 the exception.
209 
210 Example:
211 
212 class cls1 {
213 
214 int x; // private
215 
216 int y; // private
217 
218 public int meth1();
219 
220 public void meth2();
221 
222 }
223 
224 cls1.meth1() {
225 
226 if(x nequ y) {
227 
228 raise("x and y are not equal");
229 
230 }
231 
232 else {
233 
234 raise("x and y are equal");
235 
236 }
237 
238 return(x+y);
239 
240 }
241

243

252 }
